我的判断是：`v2` 最强的资产仍然是同一篇文章、不同任务的配对设计，但它的理论中心已经不稳了。新一轮文档实际上已经把问题从"Type A/B 二分法"推进到了"时间可容许性 + shortcut 类型 + 任务路径"的三件事，而 [RESEARCH_PROPOSAL_v2.md](/D:/GitRepos/LLM-Leakage-Test/docs/RESEARCH_PROPOSAL_v2.md#L88) 还把这三件事压在同一条轴上，[EXPLORE_TEMPORAL_DIRECTIONALITY.md](/D:/GitRepos/LLM-Leakage-Test/refine-logs/EXPLORE_TEMPORAL_DIRECTIONALITY.md#L6) 已经实质性推翻了这种写法。

1. 记忆分类体系应改成多维，`Type A/B` 不该再做 headline taxonomy。主 taxonomy 我建议直接改成三轴：`temporal admissibility`、`memory object`、`path role`。`memory object` 至少分 `event-outcome`、`entity prior`、`article overlap`、`provenance/source`、`regime prior`、`task-template`；`Type A/B` 只保留为功能性 shorthand。[LIT_MEMORY_TAXONOMY_NLP.md](/D:/GitRepos/LLM-Leakage-Test/refine-logs/LIT_MEMORY_TAXONOMY_NLP.md#L11) [LIT_MEMORY_TAXONOMY_COGSCI.md](/D:/GitRepos/LLM-Leakage-Test/refine-logs/LIT_MEMORY_TAXONOMY_COGSCI.md#L41)

2. `Entity prior` 应成为独立测试维度，但不要膨胀成第二主线。它往往不是严格的 look-ahead leakage，却同样会让 historical evaluation 失真，尤其在金融里会和 head entities、attention salience、target type 绑定。[LIT_MEMORY_TAXONOMY_FINANCE.md](/D:/GitRepos/LLM-Leakage-Test/refine-logs/LIT_MEMORY_TAXONOMY_FINANCE.md#L43) [LIT_MEMORY_TAXONOMY_MECHINTERP.md](/D:/GitRepos/LLM-Leakage-Test/refine-logs/LIT_MEMORY_TAXONOMY_MECHINTERP.md#L19) 我会把它定义成独立 secondary estimand：`entity-cue dependence`。实验上至少做四件事：`task × entity salience/head-tail × target_type` 分层；Tier 1 子样本做 `entity masking / ticker anonymization / neighbor-entity substitution`；Qwen familiarity 比较原文和 entity-masked 版本；authority 任务只做 restricted masking，因为全遮会把 provenance cue 一起抹掉。[DEBATE_MASKING_NLP.md](/D:/GitRepos/LLM-Leakage-Test/refine-logs/DEBATE_MASKING_NLP.md#L7)

3. CFLS 跨模板构念效度的根本解法不是"多加控制"，而是"换 primary contrast"。当前 `direct_prediction` 绑定 `semantic_reversal`，`decomposed_authority` 绑定 `provenance_swap`，所以 `Delta_CFLS` 天然混入模板显著度、顺从性、短文本改写占比和可判定性差异。[counterfactual_templates.yaml](/D:/GitRepos/LLM-Leakage-Test/config/prompts/counterfactual_templates.yaml#L69) [REVIEW_V2_CHALLENGER.md](/D:/GitRepos/LLM-Leakage-Test/refine-logs/REVIEW_V2_CHALLENGER.md#L107) 我赞成把主比较改成"同模板、不同任务"的 bridge task。项目里其实已经有 `decomposed_impact` prompt family，但它还没接到 `semantic_reversal` 上；这正好是最自然的重构方向。[task_prompts.yaml](/D:/GitRepos/LLM-Leakage-Test/config/prompts/task_prompts.yaml#L491) 在那之前，`authority` 更适合降为 secondary assay，主要靠 `evidence intrusion` 承载价值，而不是继续拿它和 direct prediction 做 headline `CFLS` 对比。

4. "泄露"定义我认为应该改成：`dependence on information not recoverable from C≤τ`。这是这轮 refinement 最有价值的升级，因为它把合法性锚定在 `point-in-time recoverability`，而不是"内容谈的是过去还是未来"。[EXPLORE_TEMPORAL_DIRECTIONALITY.md](/D:/GitRepos/LLM-Leakage-Test/refine-logs/EXPLORE_TEMPORAL_DIRECTIONALITY.md#L41) 这一定义下，论文里必须明确区分两层对象：`look-ahead leakage` 是狭义的 temporally inadmissible dependence；`text-external shortcutting` 是更宽的上位类，包含 entity prior、article overlap、template memory。于是 `CFLS` 不能再被写成"泄露测度本身"，它更像 `article-responsiveness / shortcut susceptibility`；真正更尖锐的证据是 `evidence intrusion` 和 false-outcome CPT。

5. 这些变化会让 scope 收窄，但 contribution 和可发表性会上升。最合理的 paper spine 是：一个 binary 主对比；一个 same-template bridge；一个 false-outcome CPT 核心实验；一个高精度 intrusion audit；`entity prior` 作为独立 secondary dimension；Qwen temporal/familiarity 只做 triangulation；authority/novelty/sentiment/utility 大多进 appendix。[DISCUSSION_LOG_20260406.md](/D:/GitRepos/LLM-Leakage-Test/refine-logs/DISCUSSION_LOG_20260406.md#L112) [ROUND3_EDITOR.md](/D:/GitRepos/LLM-Leakage-Test/refine-logs/ROUND3_EDITOR.md#L32) 这样改后，论文的贡献就不再是"证明 direct task 更容易泄露"这种偏显然的方向判断，而是"量化不同 task-prompt bundles 对 temporally inadmissible dependence 与其他 shortcuts 的暴露曲线，并识别 residual failure modes"。这比当前版本更像一篇能过 `Findings` 的稿子；如果没有 same-template bridge 和 clean false-outcome sign-flip，我同意现有日志的判断，最大风险仍然是 `construct validity + limited novelty`。
